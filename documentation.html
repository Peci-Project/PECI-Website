Overview

Over a decade ago, students at the University of Aveiro developed an API gateway (OPEN DATA @ UA, 2012) to centralize the institution's web services, enabling both students and faculty to easily integrate these resources into their own applications. Over time, however, the service was gradually discontinued and became obsolete.

The API UA Project revives and modernizes this gateway, making it functional, secure, and scalable. The primary goal is to centralize all of the university's web services into a single system, delivering a developer-friendly experience through a Docusaurus-based website featuring a RapidAPI-like interactive playground, while simultaneously providing a simple and intuitive management interface for system administrators.

✓ Interactive PlaygroundOnce authenticated via our secure login, users can interactively test the APIs available to them by executing live calls with custom parameters directly from the platform.
🏗️

System Architecture

To ensure maximum security and simulate a real-world production environment, our system is strictly divided into three main layers. The core services are fully isolated within a Local Private Network.

1. User Interface

The Academic Playground & Innovation acts as the portal for developers. It provides comprehensive information about the APIs and an interactive environment to test integrations and learn how to use them.

2. Local Private Network (Core)

This is the isolated heart of the system. Only the WSO2 Gateway has access to the outside network.

  • WSO2 API Layer: The sole entry point for all requests. It acts as the ultimate gateway, verifying identity through Keycloak before allowing access to any private APIs.
  • Security & Monitoring: Keycloak (IdP) manages authentication. Grafana and Loki run alongside to monitor traffic and aggregate system logs.
  • FastAPI APIs (Containerized): The modernized microservices (Parking UA, SAS, Access Points, Rss2Json) run in strictly isolated Docker containers, processing requests passed down from the gateway.

3. Backend

The external data sources consumed by our FastAPI microservices. These include legacy systems (SoapClient), external university infrastructure (WSO2 UA), and MockUp data simulating future or unavailable services.

Development Phases

The project was structured across multiple milestones to ensure continuous delivery and rigorous evaluation throughout the 2025–2026 academic year.

M1 — Plan & Objectives

We defined the project lifecycle objectives, conducted a literature review of the 2012 platform, and established the initial development plan. The core technologies (WSO2, FastAPI, Docker) were selected.

M2 — Requirements, Design & Demo

Following our pivot to a local network setup, we finalized the system architecture. We demonstrated the first functional prototypes, including the independent WSO2 gateway and the initial containerized mock services.

M3 — Prototype & Interim Evaluation

At this stage, the core microservices were fully containerized. Keycloak authentication was successfully integrated with the WSO2 gateway, and the Grafana/Loki monitoring stack was activated. We held an interim evaluation with supervisors to validate the security model.

M4 — Final Product

The interactive Docusaurus frontend was completed. Users could now log in securely and test live API calls. The four core APIs (Parking UA, SAS, Access Points, Rss2Json) were fully operational behind the WSO2 proxy.

API UA Platform

All of our modernized and containerized APIs, along with their interactive documentation and testing playgrounds, are fully centralized and available on our official platform. The core microservices include:

  • Parking UA: Provides real-time data on the availability, occupancy rates, and management of the university's various parking lots.
  • SAS (Canteen Menus UA): Delivers direct access to the menus of canteens, bars, and cafés across the campus for a defined timeline of choice.
  • Access Points: Exposes data and tracking related to the university's departments, such as network access point usage and their locations across the campus infrastructure.
  • Rss2Json: A utility service that converts RSS feeds — such as news, announcements, and events — into clean, structured JSON formats for modern application integration.

Security & Auth

Security was a paramount concern given the private nature of the university's data. Our approach guarantees that no backend service is ever exposed directly to the public web.

  • Identity Provider: We implemented Keycloak to mimic the university's central IdP. It handles user credentials and role-based access control.
  • Token Validation: The Docusaurus frontend requests an OAuth2/JWT token upon user login. This token must be passed in the Authorization: Bearer header of every API call.
  • WSO2 Policies: The gateway intercepts all requests, verifying token signatures against Keycloak before routing traffic to the internal Docker network.
ImportantAll direct access to the internal FastAPI containers is blocked by firewall rules. Requests strictly must pass through the WSO2 Gateway.
📊

Monitoring Stack

To ensure system reliability and to provide administrators with clear visibility into the gateway's performance, we implemented a robust real-time monitoring stack.

  • Grafana Dashboards: Serves as the primary interface for system administrators to track metrics, API request volumes, latency, and system health.
  • Loki for Log Aggregation: All Docker containers are configured to forward their logs directly to Loki. This centralizes logs from WSO2, Keycloak, and every FastAPI microservice.
  • Prometheus: Gathers metrics from the APIs and the host system, feeding them into Grafana to trigger alerts if performance degrades.
🐳

Docker & Deploy

The entire platform is designed for portability and isolation. Everything runs inside Docker containers orchestrated via Docker Compose.

To ensure maximum security within our local deployment:

  • Internal Network: All microservices (FastAPI), databases, Keycloak, and the monitoring tools run inside a custom Docker bridge network. They are inaccessible from the host machine or external internet.
  • Gateway Exposure: WSO2 is the only container with port mappings to the host. It acts as the ultimate reverse proxy and security checkpoint.
  • Reproducibility: Because the environment is containerized, migrating this entire setup into STIC's official infrastructure in the future requires minimal effort.
yaml
# Example of the internal network isolation in docker-compose.yml
services:
  api-sas:
    image: api-ua/sas:latest
    networks:
      - api-internal-network
    expose:
      - 8000 # Exposed only internally, NOT mapped to host

  wso2-gateway:
    image: wso2am:latest
    ports:
      - "8243:8243" # Publicly accessible
    networks:
      - api-internal-network
